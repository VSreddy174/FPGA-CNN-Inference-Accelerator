# 2. High-Level Synthesis (HLS) Core Architecture

This document describes the synthesizable C++ hardware architecture developed in Vitis HLS for the CNN inference accelerator core. It details the specific optimization directives, internal buffering mechanisms, and loop structures implemented to create a resource-balanced, high-throughput streaming pipeline.

---

## 1. Hardware Interface Directives

The accelerator maps its software controls and streaming arrays onto physical hardware interfaces utilizing specific C++ pragma configurations. The physical pins generated by the compiler conform to standard SoC interconnect methodologies:

* **Control Interface (`#pragma HLS INTERFACE s_axilite port=return`):** Maps the core execution state registers (`start`, `ready`, `idle`, and `done`) onto an AXI4-Lite peripheral control bus. This enables the master processing system to monitor and command the hardware IP block via memory-mapped software drivers.
* **Data Streaming Interfaces (`#pragma HLS INTERFACE axis port=in_stream` and `port=out_stream`):** Configures both the data ingestion and score extraction channels as high-speed AXI4-Stream master/slave interfaces. These ports handle continuous byte bursting directly from system memory-mapped domains.

*(Note: The system-level infrastructure, memory topology crossings, and AXI interconnect configurations linking these ports to the processing system are detailed in [4. Vivado SoC Integration](4_Vivado_SoC_Integration.md)).*

## 2. Layer-by-Layer Hardware Engine Implementation

The hardware computing engine executes layer-by-layer neural matrix math as a highly structured, sequential streaming dataflow.

### A. AXI-Stream Packet Parsing & Input Buffering
The input data port ingests incoming 32-bit packets from the AXI-Stream bus. To guarantee perfect byte alignment and eliminate complex bit-shifting overhead, the hardware utilizes a "padding" approach: it accepts one 32-bit DMA packet per pixel. The core extracts the lower 8 bits, immediately casting the value into an internal unsigned 8-bit register (`pixel_t`) and discarding the remaining upper 24 bits. This maps perfectly to the 32-bit width of the DMA channel while maintaining absolute alignment stability before the 28x28 image buffering and convolution begin.

### B. Convolutional Layer 1 Core (`conv_1`)
The first computing block applies 8 separate 3x3 filter channels to the buffered 28x28x1 input data.
* **Line Buffer & Sliding Window:** The core implements internal line-buffer memory arrays acting as internal FIFO queues alongside a 3x3 register matrix. This layout tracks a full neighborhood of pixels simultaneously, producing a new 3x3 sliding spatial frame every clock cycle.
* **Parallel Convolution Math:** The 9 registers of the neighborhood window feed concurrently into parallel math lines. Each line contains specialized arithmetic units executing signed 8-bit weight multiplications and accumulating into a 32-bit signed destination buffer where the `conv_1` bias is added.
* **Streaming Rectified Linear Unit (ReLU):** The accumulated 32-bit result immediately undergoes a sign-bit check. If the bit is negative, the hardware cuts the value to zero on the fly, avoiding any latency or external memory loopbacks.
* **In-Line Max Pooling:** The activated values enter a spatial downsampling node. This block monitors adjacent 2x2 pixel neighborhoods, isolates the maximum value, and discards redundant activation frames, compressing the matrix size from 26x26x8 down to a 13x13x8 feature map.

### C. Convolutional Layer 2 Core (`conv_2`)
The second stage ingests the 13x13x8 feature map from the previous block, applying 16 independent 3x3 filter windows.
* **Multi-Channel Accumulation:** Because `conv_2` features multi-channel interactions, the arithmetic units are structured to process across the 8 incoming channels. Every execution step performs a dot-product across the third dimension of the feature array.
* **Pipelined Filtering:** Similar to the first layer, this stage routes calculated 32-bit results through an embedded ReLU threshold and a second 2x2 Max Pooling module. This downsizes the spatial representation to a 5x5x16 feature map.

### D. Fully Connected Output Layer (`output_dense`)
The final block acts as the classification layer, mapping the 400 flattened features onto the 10 target digit classes (0-9).

* **Flattening Logic:** The spatial multi-channel feature maps are implicitly flattened through 1-dimensional array indexing, streaming into the dense computing array.
* **Resource-Balanced Accumulation:** To prevent a massive DSP resource explosion, the 10 output classes are calculated sequentially. The structure loop is configured with the class loop on the outside and the 400-element loop on the inside. As the hardware loops through each of the 10 digits, the pipelined inner loop streams the 400 elements sequentially, multiplying them by their respective 8-bit weights and safely accumulating them into a single reused 32-bit register per class.
* **Raw 32-bit Score Queue:** The results are combined with their respective dense biases. These raw 32-bit signed scores are queued directly into the outbound 32-bit AXI-Stream register block, bypassing the training-only Softmax activation layer to save FPGA fabric space and hardware overhead.

## 3. Memory Layout & Optimization Directives

To enforce single-cycle execution speeds across the nested mathematical loops, specific pragma instructions are injected into the C++ source file ([`cnn_accel.cpp`](../hls_core/cnn_accel.cpp)) to override standard sequential compiler choices and balance resource utilization:

* **Loop Pipelining (`#pragma HLS PIPELINE II=1`):** Applied to the innermost spatial convolution loops and the dense accumulation loop. This instructs Vitis HLS to achieve an Initiation Interval of 1 ($II=1$), forcing the hardware to ingest a new input element and calculate a new accumulation step every single clock cycle.
* **Array Partitioning (`#pragma HLS ARRAY_PARTITION dim=3 complete`):** Applied to the internal intermediate feature map buffers (`l1_pool` and `l2_pool`). By completely partitioning the 3rd dimension (channels) into individual hardware registers, memory access bottlenecks are eliminated. This allows the Layer 2 convolution multipliers to read across all 8 incoming channels simultaneously.
